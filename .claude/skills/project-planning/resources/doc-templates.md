# 기획 문서 템플릿

> 각 Phase에서 문서 작성 시 이 템플릿의 구조를 따른다.

---

## 1. project-summary 템플릿

```markdown
# {프로젝트명}

> {한줄 요약}

## 배경 및 목적

- 왜 만드는가 (문제/기회)
- 기대 효과

## 핵심 기능

우선순위 순으로 나열한다.

| # | 기능 | 설명 | 우선순위 |
|---|------|------|---------|
| 1 | ... | ... | 필수 |
| 2 | ... | ... | 필수 |
| 3 | ... | ... | 선택 |

## 대상 사용자

- 주요 사용자: ...
- 보조 사용자: ... (있다면)

## 기술적 범위

- 포함: ...
- 제외: ...
- 제약사항: ...

## 성공 기준 (선택)

- ...
```

### 작성 규칙

- 기능은 개조식으로, 한 항목에 하나의 기능만
- 우선순위는 `필수` / `선택` 2단계로 단순하게
- 기술적 범위에서 "제외" 항목을 명확히 하여 스코프 크립 방지

## 2. ux-flow 템플릿

> **문서 성격**: 프로젝트 초반 가이드 (현 상태 리포트가 아님). 디자이너가 "어떤 데이터를 어떻게 다루는지" 처음 이해하는 단계.
> **금지**: flowchart mermaid, IA 트리, 데이터 모델 관계도 (erDiagram), 화면-데이터 매트릭스, 외부 의존성 핀, "엔티티"/"M:N"/"FK"/"TTL" 같은 개발 용어, Supabase/Anthropic/Storage/Auth 같은 백엔드 용어.
> **본문 섹션 순서 강제**: 유저 시나리오 → 데이터 모델 (카드만) → UX-flow → 페이지 리스트 → 데이터 모델 활용 → 컴포넌트 리스트 → 참조.

```markdown
# {프로젝트명}. UX Flow

## 유저 시나리오

### 시나리오 1. {시나리오명}

- **사용자**: {누가}
- **목표**: {1줄}
- **흐름**: {진입점 → 핵심 단계 (동사로) → 결과}, 한 줄
- **다루는 데이터**: {데이터명 1~3개. W (생성/수정) / R (읽기) 명시}

### 시나리오 2. ...

## 데이터 모델

> 카드만 작성 (시나리오에서 등장한 데이터를 정의). UX-flow 가 그걸 단계별로 풀어쓸 수 있게. 관계도·매트릭스·외부 의존성 핀 폐기.

#### 📦 {한국어 이름} `{데이터명}`

> {1줄 설명. 일반 사용자가 읽어도 이해되게}

- **보이는 페이지**: {페이지명 1~3개}
- **만드는 사람**: {사용자 / AI / 시스템}
- **만드는 곳**: {페이지명 1개}

(데이터가 N개면 위 카드 N번 반복)

## UX-flow

> 시나리오를 데이터 관점에서 단계별로 쪼갠다. 각 단계의 페이지 · 사용자 행동 · 발생 데이터 · 결과.

### 시나리오 1 단계별

1. **{단계 이름}** ({페이지명})
   - 사용자 행동: {한 줄}
   - 발생하는 데이터: `{데이터명}` W (생성) / R (읽기) / D (수정)
   - 결과: {다음 단계로 가는 트리거}

2. ...

### 시나리오 2 단계별
...

## 페이지 리스트

| 페이지 | 경로 | 한 줄 설명 | 다루는 데이터 |
|---|---|---|---|
| Auth | `/auth` | 가입 / 로그인 | User |
| ... | ... | ... | ... |

## 데이터 모델 활용

> 이 표는 `/supabase-integration` 의 유일한 입력. 데이터명 ↔ 테이블명 1:1 매핑은 변경 시 반드시 이 표를 먼저 갱신.

| 데이터명 | 한국어 | 코드 식별자 | 예상 테이블명 | 생성 책임 페이지 |
|---|---|---|---|---|
| `Reference` | 레퍼런스 | `reference` | `reference_items` | Archive |
| ... | ... | ... | ... | ... |

## 컴포넌트 리스트

> 신규 컴포넌트만 본문에 노출 (기존 디자인 시스템에 없어 새로 만들어야 할 것). 재활용/수정은 본문 등장 금지.

| 컴포넌트 | 카테고리 | 한 줄 용도 |
|---|---|---|
| ... | ... | ... |
```

### 작성 규칙

- 시나리오: 3~5개. **4줄 양식** (사용자 / 목표 / 흐름 1줄 / 다루는 데이터 1줄).
- **UX-flow**: 시나리오 N 개에 대해 각각 단계별 서사 작성. 단계마다 페이지·행동·데이터·결과 4 항목. flowchart 다이어그램 대체.
- 페이지 리스트: 데이터 모델 활용의 "생성 책임 페이지" + 카드의 "만드는 곳" 과 글자 단위 일치.
- **데이터 모델 카드**: 4 항목 (1줄 설명 + 보이는 페이지 + 만드는 사람 + 만드는 곳). 수명·비고·"누가 읽는가" 금지. 백엔드 용어 금지.
- **데이터 모델 활용**: 5칸 모두 채움. 모르면 "(미정)" 명시 (차단 신호).
- 정합성: 카드 / 사전 / UX-flow 의 데이터명 (PascalCase) 글자 단위 일치. 사전에 없는 데이터명 등장 금지.
- **컴포넌트 리스트**: 신규만. 재활용/수정 컴포넌트는 본문 등장 금지.
- 본문/부록 분리는 성격 기준 (자연어 합의 vs 구현 디테일).

## 3. visual-direction 템플릿

```markdown
# {프로젝트명}. Visual Direction

## 톤앤매너

키워드 3~5개로 전체 방향을 정의한다.

- **키워드**: ... / ... / ... / ...
- **설명**: ...

## 컬러 방향

| 용도 | 현재 토큰 | 현재값 | 변경 방향 | 근거 |
|------|----------|--------|----------|------|
| Primary | `primary.main` | #... | ... | ... |
| Secondary | `secondary.main` | #... | ... | ... |
| Background | `background.default` | #... | 유지 |. |

## 타이포그래피 방향

| 요소 | 현재 설정 | 변경 방향 | 근거 |
|------|----------|----------|------|
| h1 | ... | ... | ... |
| body1 | ... | ... | ... |

## 간격 및 레이아웃

- spacing 기본 단위: ...
- 주요 레이아웃 패턴: ...
- 반응형 브레이크포인트 전략: ...

## 레퍼런스

사용자가 제공한 레퍼런스 이미지/사이트를 정리한다.

| # | 레퍼런스 | 참고 포인트 |
|---|---------|------------|
| 1 | {이미지 경로 or 설명} | {무엇을 참고할 것인가} |
| 2 | ... | ... |

## 변경 필요 토큰 요약

최종적으로 theme 파일에서 수정이 필요한 토큰 목록이다.

| 토큰 경로 | 현재값 | 변경값 | 적용 대상 |
|-----------|--------|--------|----------|
| `palette.primary.main` | ... | ... | ... |
| ... | ... | ... | ... |
```

### 작성 규칙

- 현재 테마 토큰값은 `mui-theme.md`에서 확인하여 정확히 기재
- "유지"하는 토큰도 표에 포함하여 의도적 유지임을 명시
- 레퍼런스 이미지는 사용자가 제공. Claude가 임의 URL 생성 금지
- 변경 토큰 요약은 구현 시 theme 파일 수정의 직접적 입력이 됨
